## Problem Statement

A developer building the *Threads of God* (TOG) UE 5.7 C++/GAS remake has a complete, compiling damage-and-reaction spine in C++ — two modules (`TOGCore`, `TOG`), 19 native gameplay tags, an `FTOGGameplayEffectContext` with bit-packed `NetSerialize`, `UExecCalc_Damage` (health + poise channels), the cross-actor hit-react keystone in `UTOGAttributeSet::PostGameplayEffectExecute`, dual-path GAS init (player ASC on PlayerState/Mixed, enemy ASC on Character/Minimal), data-driven attribute init via `UTOGAbilitySystemLibrary::InitializeDefaultAttributes`, and tag-driven input forwarding — but **none of it runs in Play-In-Editor (PIE)**. Every C++ verb is generic and waits on data the developer has not yet authored: there are no GameplayEffects, no concrete abilities, no DataTables, no Character/PlayerState/GameMode Blueprints, no input config, and no test level. The C++ is "green" but the game is not yet playable; a developer cannot press an attack input and watch an enemy take damage and react.

Compounding this: the UE 5.7 remake has **no Python remote-execution bridge** (unlike the UE 5.4 reference project), so every remaining asset must be hand-authored in the editor — there is no automation and no automated test net. The work is highly order-dependent (e.g. a default-attributes GE that seeds a current value before its Max value spawns the pawn dead-on-arrival), and several gameplay-design decisions are deliberately parked in a review docket pending the first time PIE forces the call. The developer needs a single sequenced specification that turns the green C++ keystone into a playable vertical slice without forking away from idiomatic GAS or silently mis-wiring a seam.

## Solution

Author, in a fixed dependency order, the minimum set of editor assets that lets the existing C++ pipeline execute end-to-end in PIE, then build the first two concrete abilities on top of it. From the developer's perspective:

1. **Data-driven spawn (D1 assets).** The player and the boss spawn with Health/Stamina/Poise read from per-class `GE_DefaultAttributes_*` effects, resolved through a `DA_CharacterClassInfo` data asset on a `BP_TOGGameMode`. A missing or mis-wired data asset fails *loudly* (`ensureAlwaysMsgf`), never silently at a hardcoded 100 HP (that fallback was deleted in D1).
2. **Damage GE (D2).** An instant `GE_Damage` whose only Execution is `UExecCalc_Damage` lets any SetByCaller-stamped damage spec drain into `IncomingDamage`/`IncomingPoiseDamage` and through `PostGameplayEffectExecute` into Health/Poise.
3. **First attack ability (D3).** A `GA_Attack` BP subclass of `UTOGDamageGameplayAbility` plays a montage, waits on a montage `AN_MontageEvent`, resolves a target, and applies `GE_Damage` — the first complete input → ability → montage notify → ExecCalc → attribute-change path, client-predicted and server-authoritative.
4. **Hit-reaction ability (D4).** A BP ability triggered by the native `Event.Reaction.HitReact` event reads the per-hit `FTOGGameplayEffectContext` riding in the event payload and plays a directional reaction montage; a poise-broken (`State.Staggered`) victim plays a heavier variant.

The result is a playable PIE slice that exercises the keystone: a player swing damages an enemy, the enemy reacts directionally, accumulated poise damage staggers it, and a lethal hit routes through `ICombatInterface::Die()`.

## Implementation Decisions

**Modules built/modified**
- Both runtime modules already exist and compile: `TOGCore` (native tags `FTOGGameplayTags`, structs `FTOG_TaggedMontage`/`FTOGComboRow`, interfaces `ICombatInterface`/`IPlayerInterface`/`ISaveInterface`/`ITargetableInterface`) and `TOG` (gameplay layer). This PRD authors **editor assets that consume existing C++ symbols** — no new C++ modules, and minimal-to-no new C++ for D1–D4 (D2's `GE_Damage` is asset-only; D3/D4 are BP abilities + DataTables).

**Interfaces / seams already present that the assets bind to (highest existing seams, preferred over new ones)**
- **Attribute-init seam:** `UTOGAbilitySystemLibrary::InitializeDefaultAttributes(WCO, ETOGCharacterClass, Level, ASC)` → resolves `ATOGGameModeBase::CharacterClassInfo` → `UTOGCharacterClassInfo::GetClassDefaultInfo` (uses `FindRef`, not `FindChecked`) → applies the per-row Instant GE. Two `ensureAlwaysMsgf` loud-failures guard a missing data asset and a missing GE. Called authority-gated from `ATOGPlayerCharacter::InitAbilityActorInfo` (before `GrantStartupAbilities`) and `ATOGEnemy::InitAbilityActorInfo`.
- **Damage-apply seam:** `UTOGAbilitySystemLibrary::ApplyGameplayEffectToTarget(FTOG_DamageEffectParams)` builds a `FTOGGameplayEffectContext`, stamps custom fields, and assigns SetByCaller magnitudes keyed on `DamageTypeTag` (health) and `Damage.Poise` (poise). The damage type tag *is* the SetByCaller key (Aura idiom).
- **ExecCalc seam:** `UExecCalc_Damage::Execute_Implementation` sums health damage types (currently hardcoded `Damage.Physical` + `Damage.Fire`) into the `IncomingDamage` meta and routes `Damage.Poise` into `IncomingPoiseDamage`. No attribute captures yet (C1 pass-through; armor/resistance deferred).
- **Meta-drain / keystone seam:** `UTOGAttributeSet::PostGameplayEffectExecute` drains `IncomingDamage`→Health (then on fatal → `State.Dead` loose tag + `ICombatInterface::Die(DeathImpulse)`; on non-fatal → `SendGameplayEventToActor(victim, Event.Reaction.HitReact, payload)`) and `IncomingPoiseDamage`→Poise (on break → `State.Staggered` once). The non-fatal event's `Instigator` is the effect causer/avatar pawn (not the PlayerState ASC owner). A `State.Dead` guard prevents re-processing a corpse.
- **Ability base seam:** `UTOGDamageGameplayAbility::MakeDamageEffectParamsFromClassDefaults(TargetActor)` and `ApplyDamageToTarget(TargetActor)`; class-default `DamageEffectClass`, `DamageType`, `Damage` (FScalableFloat). Base `UTOGGameplayAbility` carries `StartupInputTag`.
- **Montage-event seam:** `UAN_MontageEvent::Notify` sends a gameplay event tagged by its `EventTag` to the mesh owner; payload carries `Instigator` only (targeting is the ability's job).
- **Input seam:** `UTOGInputComponent::BindAbilityActions` (template) iterates `UTOGInputConfig::AbilityInputActions` and binds Started/Completed/Triggered to the controller's `AbilityInputTagPressed/Released/Held`, which forward to `UTOGAbilitySystemComponent`. The ASC matches abilities by `GetDynamicSpecSourceTags().HasTagExact(InputTag)` (UE 5.7 API), de-duplicating replicated input events to the latest instance.
- **Custom GE context seam:** `FTOGGameplayEffectContext` (`bIsParried`/`bIsGuardBroken`/`bIsFinisherEligible`/`HitstopMagnitude`/`DamageTypeTag`), `GetScriptStruct` returns the derived struct, bit-packed `NetSerialize` (12 bits), registered via `UTOGAbilitySystemGlobals::AllocGameplayEffectContext`. Accessors are type-guarded through the `TOGCtx`/`TOGCtxMutable` helpers (D0.2 done).

**Architectural decisions (already encoded in code, governing the asset authoring)**
- ASC placement: player ASC on `ATOGPlayerState` (Mixed replication, `NetUpdateFrequency` bumped to 100 Hz); enemy ASC on `ATOGEnemy` (Minimal replication). Dual init path: player via `PossessedBy`/`OnRep_PlayerState`; enemy via `BeginPlay`.
- Flat attribute set (Health/MaxHealth, Stamina/MaxStamina, Poise/MaxPoise + two meta attributes) — no Aura-style Primary/Secondary/Vital cascade — so a *single* Instant GE seeds all base values per class. This drives the **MAX-before-current modifier ordering** requirement: `PreAttributeBaseChange` (the `const` override, D0.3 done) and `PreAttributeChange` both clamp current to `[0, Max]`, so a Max modifier must precede its current modifier or the pawn spawns at 0. `Override` modifiers do not avoid this; only ordering does.
- Two-archetype `ETOGCharacterClass { Player, Boss }` keyed map (trimmed from Aura's three classes); no XP economy, no StartupAbilities array on the row (abilities granted separately via `GrantStartupAbilities`).
- Death is a character-level concern through `ICombatInterface::Die()`, **not** a reaction GameplayAbility. Hit reactions are BP abilities triggered by a native event; C++ only dispatches.
- Guiding principle throughout: build TOG (the game) with idiomatic GAS architecture; where Aura diverges from canonical GAS, prefer canonical GAS. Aura is the reference, not a structure to mirror.

**Schema / data-asset contracts**
- `UTOGCharacterClassInfo`: `TMap<ETOGCharacterClass, FTOGCharacterClassDefaultInfo>` where each row holds one `TSubclassOf<UGameplayEffect> DefaultAttributes`.
- `FTOG_TaggedMontage`: `{ Montage, MontageTag, SocketTag, ImpactCueTag }` — the montage's `MontageTag` must equal the `AN_MontageEvent.EventTag` and the ability's `WaitGameplayEvent` tag.
- `FTOGComboRow` (FTableRowBase): `{ Montage (soft), ComboWindowStart, ComboWindowEnd=0.3, DamageTypeTag, BaseDamageMultiplier=1 }` — available for the combo DataTable (D3+).
- `FTOG_DamageEffectParams`: carries `BaseDamage`, `PoiseDamage`, `AbilityLevel`, `DamageTypeTag`, `HitstopMagnitude` (default 0.1), source/target ASCs, and `DamageGameplayEffectClass`.

**Open decisions to resolve at the forcing step (from review docket / Phase-D triage)**
- Poise tag taxonomy (`Damage.Poise` under `Damage.*` vs a `Data.*` root; health types currently hardcoded vs looped) — decide at D2.
- Verify both `PostGameplayEffectExecute` branches fire on a combined hit — D2 PIE gate.
- Single-reaction-event policy (pure-poise attack sends no event; stagger-vs-event ordering) — D4.
- Finisher-eligibility source — D5 (out of this PRD's core scope but flagged).
- Death impulse: placeholder `DeathImpulseStrength = 4000.f` constant vs promoting `DeathImpulse` into the context (Aura approach) — D4/D5.

**Configuration contract (a grounding gap to close)**
The editor-work checklist references `Config/DefaultEngine.ini` (`AssetManagerClassName`, `GlobalDefaultGameMode`), `Config/DefaultGame.ini` (`AbilitySystemGlobalsClassName`), and `Config/DefaultInput.ini` (`DefaultInputComponentClass`) with "verified line N", but **no `Config/` directory and no `Content/` directory exist in this repository checkout** (`.gitignore` does not exclude `Config/`). These ini overrides are mandatory: without `DefaultInputComponentClass`, `ATOGPlayerController::SetupInputComponent`'s `CastChecked` hard-crashes on Play; without the AssetManager/GAS-globals overrides, native tags and the custom GE-context factory don't register. Authoring assets must include verifying these files exist (and committing them) before the D1 PIE gate.

## Testing Decisions

**What makes a good test here:** test *external behavior at a seam*, not implementation details. The seams to assert at are the PIE-observable outcomes of the keystone — attribute values after a GE application, the event dispatched to a victim, and the montage that plays — never the internal SetByCaller bookkeeping or the NetSerialize bit layout. Because there is **no automated test net** and **no Python remote-exec on the 5.7 remake**, testing for D1–D4 is manual PIE smoke-checking at the marked gates, kept honest by small changes and the critical-agent review gates (`cpp-pro`, `game-developer`, `/code-review` fan-out) — each subagent claim verified against ground truth (a prior session had a subagent fabricate RAM figures and a reviewer flag a non-existent compile error; the build result line was the tell).

**Modules / behaviors tested (PIE gates):**
- **D1 spawn gate:** player + enemy spawn with data-driven Health/Stamina/Poise; the negative test (clear `CharacterClassInfo` or a row GE) must fail loudly via `ensureAlwaysMsgf`, not silently. Confirm player Level sources from PlayerState, enemy from its pawn.
- **D2 damage gate:** applying `GE_Damage` drops Health *and* Poise; explicitly confirm both `PostGameplayEffectExecute` branches fire on one combined hit (docket #2).
- **D3 attack gate:** attack input → montage → at the notify, enemy Health drops by the GE magnitude (observe via `showdebug abilitysystem` or an `OnHealthChanged` debug print). Prediction check: `Play As Client` shows the change immediately and matches the server with no rollback under normal latency.
- **D4 reaction gate:** a hit produces a *directional* reaction montage on the victim; a poise-broken (Staggered) victim plays the heavier variant; a lethal hit routes through `Die()` and ends behavior cleanly without sending a reaction event to the corpse.
- **Three must-pass end-to-end scenarios:** (1) cross-actor pairing (attacker swing → correct directional victim reaction); (2) poise-break → stagger flips `State.Staggered` and selects the heavier variant; (3) lethal → `Die()`.

**Build verification (every connected C++ change):** UBT `Result: Succeeded` line **plus** a relinked DLL whose mtime is newer than the edited source — never the wrapper exit code. clangd "CoreMinimal.h not found" noise is not a real error (clangd lacks UE include paths); only UBT is authoritative.

**Prior art for the tests:** the Step-4 spike (`docs/planning/Step4-Spike-EditorHandoff.md`) is the canonical worked example of this manual PIE flow (GE → ability graph → `AN_MontageEvent` → `FTOG_TaggedMontage` row → BP defaults → test level → PIE damage + prediction check). The Aura reference project supplies the BP-authoring shapes (`Aura-BP-Authoring-Recipes.md`, `Aura-GAS-Patterns.md` §13 Lift Kit). The 5.4 original is read-only ground truth for reaction directions, montage notify timings, and DataTable values via the **ue-inspector** subagent — read it before authoring the 5.7 equivalent, never guess. A characterization-test pass (e.g. `code-modernization`'s `test-engineer`) is a candidate once the abilities stabilize, but is not part of this PRD.
